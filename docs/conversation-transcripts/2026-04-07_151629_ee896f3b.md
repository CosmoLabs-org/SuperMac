Dispatches N parallel Opus agents (3-10+) to deeply analyze any codebase. Produces persistent intelligence artifacts in docs/audit/. Works on ANY project — from mature repos to raw artifact folders.

## Quick Start

```bash

/audit --focus quality          # Code Quality + Core Logic only
/audit --focus growth           # Frontend/UX + Competitive + SEO
```

## Quality Standard

**The purpose is to IMMORTALIZE intelligence.** Every agent scan gathers deep context, insights, and actionable advice about the codebase — architecture decisions, improvement opportunities, risk areas, patterns worth preserving or fixing. That intelligence MUST be preserved in `docs/audit/` as durable project artifacts. Every unwritten file is wasted computation.

**What must be preserved:**
- Every dispatched agent's full analysis → individual report file on disk (insights, scores, recommendations)
- Cross-agent synthesis → architecture.md, patterns.md, entry-points.md, risk-map.md (connecting findings across agents)
- Actionable advice → upgrade-plan.md (phased recommendations for every area that could be improved)
- Project brief → brief.md (condensed intelligence for future sessions to load instead of re-auditing)
- Machine-readable data → scorecard.json, context.json, metrics.json
- Executive summary → README.md

**13+ files minimum** for a standard audit. But the goal is not a file count — it's preserving every valuable insight. If an agent found something unexpected, give it its own section or file. If the project has specific areas needing attention (a single feature, a subsystem, a migration path), the synthesis should address those specifically, not just generically.

**Aim higher**: Don't stop at filling templates. The best audits produce intelligence that changes how the team thinks about the codebase. Cross-reference findings, surface non-obvious connections, and write concrete improvement advice with file paths and priorities.

## Detailed Security Analysis: SuperMac v2.1.0

### Executive Summary

SuperMac is a modular macOS CLI tool consisting of a main dispatcher (`bin/mac`), a utilities library (`lib/utils.sh`), and nine feature modules. The codebase is well-structured with consistent patterns and no use of `eval` or `exec`. However, it contains several significant security weaknesses centered around its installation process, privilege escalation patterns, and lack of input sanitization before passing data to system commands. The most severe issues are the unverified `curl | bash` install pipeline and the blanket `/tmp/*` deletion during system cleanup.

### 1. Command Injection Analysis (Score: 45/100)

The codebase avoids the most dangerous patterns -- there are zero `eval` statements and zero `exec` calls. This is a significant positive. However, command injection risks remain through string interpolation into command arguments.

**osascript String Interpolation**

In `display.sh:63`, user-supplied brightness values are interpolated into an osascript command:

```bash
osascript -e "tell application \"System Events\" to set brightness of first item of (displays whose built in is true) to $decimal"
```

While `is_in_range` validates the input is numeric (0-100), the pattern of interpolating values into osascript strings is fragile. If the validation were ever bypassed or the code path changed, this becomes a direct injection vector. A safer approach is to pass values through environment variables or use heredoc-style osascript invocations.

**Network Name Injection**

In `wifi.sh:331-334`, the network name from user input flows directly to `networksetup`:

```bash
networksetup -setairportnetwork "$interface" "$network_name"
```

Network names can contain arbitrary Unicode, quotes, and special characters. While `networksetup` handles its own argument parsing, feeding it unvalidated names is a risk if the tool is ever used in automated scripts.

**ANSI Injection via Terminal Output**

The `print_success`, `print_error`, `print_info`, and `print_warning` functions in `utils.sh` all use `echo -e` with interpolated data. Since the `echo -e` flag interprets backslash escape sequences, any user-controlled data (process names from `dev.sh`, network names from `wifi.sh`, device names from `audio.sh`) could contain ANSI escape sequences. An attacker controlling a process name (e.g., via a malicious binary) could inject terminal escape sequences to manipulate output, hide malicious activity, or potentially exploit OSC 52 clipboard sequences.

**find Command with Unsanitized Input**

In `dock.sh:350`, the application name is used directly in a find command:

```bash
app_path=$(find /Applications -maxdepth 2 -iname "*$app_name*.app" -type d | head -1)
```

If `app_name` contains characters like `"` or `;`, the find pattern could match unintended files.

### 2. Privilege Escalation Analysis (Score: 30/100)

The codebase uses `sudo` extensively in the `network` and `system` modules. The privilege escalation patterns are concerning for several reasons.

**Silent sudo Usage**

In `system.sh:155-158`, the cleanup function tests for sudo access non-interactively (`sudo -n true`) and then proceeds to run destructive sudo commands:

```bash
if [[ "$EUID" -eq 0 ]] || sudo -n true 2&gt;/dev/null; then
    sudo find /var/log -name "*.log" -mtime +7 -delete 2&gt;/dev/null || true
    sudo find /private/var/log -name "*.log" -mtime +7 -delete 2&gt;/dev/null || true
```

The `sudo -n` flag means no password prompt is shown. If a user has an active sudo session (common during system maintenance), these commands execute silently. The user sees "Cleaning system logs" but is not explicitly asked to authorize each sudo operation.

**Destructive Network Reset**

The `network_reset` function in `network.sh:312-332` is the most dangerous privileged operation:

```bash
sudo rm -f /Library/Preferences/SystemConfiguration/NetworkInterfaces.plist
sudo rm -f /Library/Preferences/SystemConfiguration/preferences.plist
sudo launchctl unload /System/Library/LaunchDaemons/com.apple.networkd.plist
sudo launchctl load /System/Library/LaunchDaemons/com.apple.networkd.plist
```

This deletes the system's network configuration files with no backup. If the operation is interrupted or the subsequent launchctl commands fail, the Mac is left with no network configuration. There is no recovery path documented or implemented.

**Font Cache Clearing**

`system.sh:178` runs `sudo atsutil databases -remove` without any specific consent prompt for this privileged operation. It happens as part of the general "cleanup" flow.

### 3. Insecure Install Analysis (Score: 15/100)

The installation process is the weakest security point in the entire project.

**curl | bash with No Verification**

The README and install.sh document the installation method as:

```bash
curl -fsSL https://raw.githubusercontent.com/CosmoLabs-org/SuperMac/main/install.sh | bash
```

This pattern is inherently insecure because:
1. The downloaded script is executed before the user can review it
2. The URL points to the `main` branch, not a pinned commit or tag
3. There is no checksum verification of the downloaded content
4. There is no GPG signature verification
5. GitHub serves content over CDN; a CDN compromise or DNS hijack could serve malicious content

**No Integrity Checks on Downloaded Files**

The `download_file` function (`install.sh:112-135`) downloads files via curl or wget and writes them directly to disk:

```bash
curl -fsSL "$url" -o "$output" 2&gt;/dev/null
```

There is no SHA256 hash check, no signature verification, no content-length validation. An attacker who can modify the download (MITM with a compromised CA, DNS poisoning, GitHub repo compromise) can replace any module file with malicious content.

**Shell Configuration Modification**

The installer modifies the user's shell configuration (`install.sh:269-272`):

```bash
echo '' &gt;&gt; "$shell_config"
echo '# Added by SuperMac installer (CosmoLabs)' &gt;&gt; "$shell_config"
echo 'export PATH="$HOME/bin:$PATH"' &gt;&gt; "$shell_config"
```

This happens without specific user consent for the modification. The `check_path_configured` function only checks if the exact string already exists, but the modification itself is not gated behind a prompt.

**PATH Prepending Security Risk**

The PATH modification prepends `$HOME/bin` to PATH, meaning any executable in `~/bin` takes precedence over system commands. Combined with the tool's reliance on PATH-based command lookup (see Dependency Security), this creates an escalation vector.

### 4. Path Traversal Analysis (Score: 55/100)

The codebase handles paths in a mostly safe manner, but there are specific concerns.

**Custom Screenshot Location**

`screenshot.sh:70-77` accepts arbitrary directory paths:

```bash
if [[ -d "$location" ]]; then
    target_path="$location"
```

This is passed to `defaults write com.apple.screencapture location`. While defaults write is safe, the path is used as-is without normalization or symlink resolution.

**HTTP Server Directory**

`dev.sh:191-224` serves an arbitrary directory via Python's HTTP server:

```bash
cd "$directory" || return 1
python3 -m http.server "$port"
```

A user could inadvertently serve sensitive directories (e.g., `mac dev serve /`) if not careful. There is no warning for paths outside the current working directory or home directory.

**File Reveal**

`finder.sh:152-169` passes user-supplied paths to `open -R`. While `open` is a system command that handles its own security, the path is not resolved against symlinks before use.

### 5. Secrets and Credentials Analysis (Score: 80/100)

This is the strongest security area. The codebase contains:

- No hardcoded API keys, tokens, or passwords
- No `.env` files or credential files
- The `config/config.json` contains only settings (screenshot location, volume steps, color output preferences) -- nothing sensitive
- Password generation (`dev_password`) uses `openssl rand` or `/dev/urandom` with proper entropy sources

The one concern is that generated passwords are echoed to the terminal and copied to the clipboard, where they may persist in terminal scrollback, shell history, or clipboard managers.

### 6. Dependency Security Analysis (Score: 40/100)

The tool relies on many macOS system commands without absolute path qualification:

- `osascript` -- AppleScript execution
- `defaults` -- macOS preferences management
- `networksetup` -- network configuration
- `system_profiler` -- hardware/software information
- `lsof` -- port and file handle inspection
- `killall` -- process termination
- `find` -- file system traversal
- `curl`/`wget` -- network downloads
- `jq` -- JSON parsing (optional)
- `SwitchAudioSource` -- audio device management (optional, Homebrew)
- `dockutil` -- dock management (optional, Homebrew)
- `bc` -- arithmetic calculations

None of these are called with absolute paths. Since the installer prepends `~/bin` to PATH, a malicious executable placed in `~/bin` with any of these names would be executed instead of the system version. This is a real attack vector for users who install SuperMac via the recommended method.

The optional dependencies (`SwitchAudioSource`, `dockutil`) from Homebrew add additional trust requirements. If these packages are compromised or typo-squatted, the tool would execute malicious code.

### 7. Race Condition Analysis

The codebase does not exhibit significant TOCTOU (time-of-check-time-of-use) vulnerabilities in the traditional sense, since most operations are single-step. However:

- `system.sh:155-158` checks for sudo access (`sudo -n true`) and then uses sudo later. The sudo session could expire between the check and the operation.
- `screenshot.sh:406-416` generates predictable filenames in `/tmp`, creating a window for symlink attacks.
- `dev.sh:337-342` checks if a file exists, then creates a temp file alongside it. In a shared directory, the temp file path is predictable.

### 8. Temp File Handling Analysis (Score: 25/100)

Temp file handling is one of the weakest areas.

**Predictable Screenshot Filenames**

`screenshot.sh:406-416`:

```bash
screencapture -x "/tmp/screenshot-$(date +%s).png"
screencapture -i "/tmp/screenshot-$(date +%s).png"
screencapture -i -w "/tmp/screenshot-$(date +%s).png"
```

The filename is entirely predictable based on the current timestamp. An attacker monitoring `/tmp` can create symlinks at the expected path that point to sensitive files (e.g., `~/.ssh/authorized_keys`), and the screencapture command will overwrite the target.

**JSON Formatting Temp Files**

`dev.sh:337-338`:

```bash
jq '.' "$file" &gt; "${file}.formatted"
```

The temporary file `${file}.formatted` is predictable and created alongside the original file. In a shared directory, this is exploitable.

**No mktemp Usage Anywhere**

The codebase never uses `mktemp` for temporary file creation. All temporary files use predictable naming schemes.

### 9. Unsafe eval/exec Analysis

This is a positive finding: there are zero `eval` statements and zero `exec` statements in the entire codebase. All command dispatch is done through `case` statements and direct function calls, which is the correct approach.

### Summary of Recommendations (Priority Order)

1. **Install Security**: Pin downloads to commit SHAs, add SHA256 checksums, remove the `curl | bash` recommendation or add a verification step
2. **Temp Files**: Replace all predictable temp file paths with `mktemp`
3. **Destructive Operations**: Add explicit per-operation consent for sudo commands, create backups before deleting system files
4. **Input Sanitization**: Validate all user inputs against strict patterns before passing to osascript, networksetup, find, or echo -e
5. **Command Paths**: Use absolute paths for system commands or verify command integrity at startup
6. **/tmp Cleanup**: Never blanket-delete `/tmp/*`; clean only user-owned files

## SuperMac Architecture Audit -- Detailed Analysis

### 1. Project Overview

SuperMac v2.1.0 is a modular Bash CLI for macOS system management. It uses a dispatcher pattern where `bin/mac` routes commands to lazy-loaded module files in `lib/`. The project has 9 functional modules covering Finder, WiFi, networking, system info/maintenance, developer tools, display settings, dock management, audio control, and screenshots, backed by a shared utilities library. Total codebase is approximately 4,800 lines of functional code (excluding duplicates).

### 2. Dependency Graph and Module Coupling

```
                    bin/mac (dispatcher, 349L)
                        |
                    sources utils.sh
                        |
            +-----------+-----------+
            |                       |
        utils.sh (589L)      config.json (65L, unused)
            |
    +---+---+---+---+---+---+---+---+---+
    |   |   |   |   |   |   |   |   |   |
   find wifi net sys dev dsp dock aud ssc
   313L 507 484 541 610 428 573 519 590

   Cross-module:
   network ---&gt; wifi (optional, runtime probe)
```

The dependency graph is essentially flat. Every module depends only on `utils.sh` and is otherwise independent. This is a strong architectural choice. The single cross-module dependency (network optionally calling wifi functions via `declare -f` probing at `network.sh:180`) is handled gracefully with a fallback path. There are no circular dependencies.

**Coupling score: 75/100** -- Loose coupling is achieved. The only concern is that utils.sh acts as a shared god-file, meaning any change to it potentially affects all modules. The `export -f` block at `utils.sh:575-587` exports 47 functions into the global namespace, which is a form of implicit coupling.

### 3. Separation of Concerns

Each module generally owns one macOS domain:

| Module | Domain | Lines | Verdict |
|--------|--------|-------|---------|
| finder | File visibility &amp; Finder | 313 | Clean |
| wifi | WiFi connectivity | 507 | Clean |
| network | IP, DNS, network diag | 484 | Mostly clean |
| system | System info &amp; maintenance | 541 | Cleanup scope too broad |
| dev | Dev tools &amp; encoding | 610 | Two unrelated domains |
| display | Brightness &amp; appearance | 428 | Clean |
| dock | Dock positioning &amp; config | 573 | Clean |
| audio | Volume &amp; device switching | 519 | Clean |
| screenshot | Screenshot settings | 590 | Clean |

The dev module is the worst offender. Lines 30-315 handle process and server management (kill-port, list-ports, servers, localhost, serve, processes, cpu-hogs, memory-hogs). Lines 321-461 handle encoding and generation (json-format, base64-encode, base64-decode, uuid, password). These are two completely unrelated domains crammed into one module because both appeal to "developers." The process management commands overlap significantly with the system module's `system_processes()` function, which does nearly the same thing (`system.sh:388-413` vs `dev.sh:230-271`).

The system module's `system_cleanup()` function at lines 109-185 also conflates too many concerns: it handles cache cleanup, download pruning, trash emptying, system log rotation, Safari cache clearing, temp file removal, and font cache clearing in a single function with a single confirmation prompt. The `rm -rf /tmp/*` at line 173 is particularly concerning.

**Separation score: 70/100** -- Generally good domain boundaries, but dev module has a split personality and system cleanup needs tighter scoping.

### 4. Extensibility Analysis

Adding a new module requires touching at least 3 files manually:

1. Create `lib/newmod.sh` with the required pattern (dispatch, help, search, action functions)
2. Add entry to `CATEGORIES` dict in `bin/mac` at line 47
3. Add to test module lists in `tests/test.sh` at lines 81, 131, 160, 196

Optionally:
4. Add global shortcuts to `GLOBAL_SHORTCUTS` in `bin/mac` at line 60
5. Add aliases to `config.json`
6. Add category entry to `config.json`

There is no auto-discovery mechanism. The `get_available_modules()` function at `utils.sh:524-529` is capable of scanning `lib/*.sh`, but it is never called by the dispatcher. The dispatcher relies entirely on the hardcoded `CATEGORIES` associative array.

There is no module template file, no scaffolding script, no `mac create-module` command. A new contributor would need to read an existing module and manually replicate the pattern.

The module contract itself is well-defined and consistent:
- `{name}_dispatch(action, args...)` -- routes actions to handler functions
- `{name}_help()` -- prints formatted help
- `{name}_search(term)` -- returns matching commands
- `{name}_{action}(args...)` -- individual action handlers

This contract could easily be formalized and auto-validated.

**Extensibility score: 55/100** -- The contract is clean but the registration process is manual and error-prone. No template, no auto-discovery, no scaffolding.

`utils.sh` at 589 lines is a catch-all utility file containing:

| Output functions | 88-206 | 15 print_* functions | Yes -&gt; output.sh |
| Validation | 212-248 | 5 check/is functions | Yes -&gt; validation.sh |

The file serves as both a "standard library" and a "runtime infrastructure" module. The color constants alone take 60 lines. The box-drawing characters take 20 lines. These are never going to change and add noise to the file.

On the positive side:
- The `load_module()` function provides clean lazy-loading
- The `get_config()` function provides a config abstraction (even if unused)
- Error handling with `die()` and `cleanup()` is properly structured
- The `print_*` function hierarchy (success/error/info/warning/debug) is well-designed

The `export -f` block at lines 575-587 exports 47 functions. This is heavy but necessary in Bash since subshells created by pipes and command substitution do not inherit functions.

**Shared utilities score: 65/100** -- Functional but bloated. The single-file approach makes it hard to navigate and modify. The config reading infrastructure exists but is completely unused by modules.

### 6. File Organization Assessment

This is the weakest aspect of the project. The file tree has severe duplication:

```
SuperMac/
  bin/mac          &lt;-- canonical dispatcher
  lib/*.sh (10)    &lt;-- canonical modules
  config/config.json &lt;-- canonical config
  tests/test.sh    &lt;-- canonical tests

  mac              &lt;-- duplicate of bin/mac
  *.sh (13)        &lt;-- duplicates of lib/*.sh + install + setup + test
  config.json      &lt;-- duplicate of config/config.json
```

Every file is confirmed identical via diff. The root directory contains 14 duplicated files totaling approximately 200KB. This creates confusion about which file is authoritative, bloats git history, and makes it unclear where to make edits. A contributor might edit `finder.sh` at the root and not realize the running code uses `lib/finder.sh`.

Additionally:
- `install.sh` at the root (356 lines) is not in `bin/` -- inconsistent
- `setup.sh` at the root (436 lines) is not in `bin/` -- inconsistent
- `test.sh` at the root (509 lines) duplicates `tests/test.sh` -- wait, it does not. Let me verify.

The `tests/` directory contains `test.sh`, and there is also a root `test.sh`. The root `install.sh` and `setup.sh` have no counterparts in `bin/` (the bin/ only contains `mac` and `install.sh`). This inconsistency means the file organization story is incomplete even for non-duplicated files.

**File organization score: 35/100** -- The canonical structure (bin/, lib/, config/, tests/) is well-conceived but completely undermined by identical root-level copies. This is the most critical organizational issue.

### 7. Configuration Architecture

The `config.json` file at 65 lines defines:
- `version`: "2.1.0"
- `command_name`: "mac"
- `default_settings`: screenshot_location, screenshot_format, volume_step, brightness_step, color_output, debug_mode, auto_update_check
- `aliases`: ip, cleanup, restart-finder, kp, vol, dark, light
- `categories`: 9 entries with enabled flags and descriptions
- `user_preferences`: favorite_commands, command_history, last_used

The configuration system has a critical implementation gap:

**What config.json declares vs. what the code actually uses:**

| Config field | Declared | Consumed at runtime |
|-------------|----------|-------------------|
| default_settings.screenshot_location | Yes | No -- screenshot module hardcodes Desktop |
| default_settings.screenshot_format | Yes | No -- screenshot module reads from defaults |
| default_settings.volume_step | Yes | No -- audio module hardcodes step of 10 |
| default_settings.brightness_step | Yes | No -- display module has no step concept |
| default_settings.color_output | Yes | No -- colors are determined by terminal check |
| default_settings.debug_mode | Yes | No -- debug is set via CLI flag |
| default_settings.auto_update_check | Yes | No -- no update mechanism exists |
| aliases | Yes | No -- hardcoded in bin/mac GLOBAL_SHORTCUTS |
| categories.*.enabled | Yes | No -- all categories are always enabled |
| user_preferences | Yes | No -- never read or written |

The `get_config()` function at `utils.sh:429-441` is the sole bridge between config.json and the code, but zero modules call it. The configuration file is effectively documentation that the code ignores.

**Configuration score: 50/100** -- The schema is well-designed, but it is completely disconnected from the runtime. The config file is decorative.

### 8. Plugin/Extension System

There is no plugin architecture. The `plugins/` directory exists but contains only a symlink to an external project (`ClaudeCodeSetup/plugins/internal`) and an 85-byte `registry.json` -- these appear to be artifacts from a different tool, not related to SuperMac's module system.

Third-party extensions would currently require:
1. Creating a new `.sh` file in `lib/`
2. Manually editing `bin/mac` to add the category
3. Manually editing `config.json` to add the category entry
4. Manually editing `tests/test.sh` to add test coverage

There is no way to "drop in" a module without modifying core files.

### 9. Scalability Assessment

The current architecture will face specific challenges as it grows:

**Startup latency**: Currently, only `utils.sh` is loaded at startup (~589 lines sourced). Modules load on demand. This is good and will scale well to 20+ modules since only the invoked module is sourced.

**Command routing**: The dispatcher uses a flat case statement in each module's `_dispatch()` function. As action count grows per module, these case statements will become unwieldy. The dev module already has 13 actions in its dispatcher.

**Namespace pollution**: Every module sources into the global namespace. The finder module alone defines 11 functions, and dev defines 18. With 20 modules, there would be 200+ global functions. The `export -f` block already exports 47 functions. Bash has no module scoping, so collisions become likely.

**Test scalability**: The test suite hardcodes module lists in 4 separate locations. Adding a module means updating all 4. There is no test discovery mechanism.

**Search scalability**: Each module's search function uses hardcoded substring matching against fixed keywords. This does not scale to large command sets and produces inconsistent results.

**Configuration scalability**: The config.json is read via `jq` for each `get_config()` call. If modules start using it heavily, this becomes a performance bottleneck (spawning a jq process per config read).

### 3. Test Quality Analysis

#### 3.1 Assertion Quality

The test suite uses a single assertion pattern for all functional tests:

```bash
if "$BIN_DIR/mac" &lt;command&gt; &gt;/dev/null 2&gt;&amp;1; then
    test_success "&lt;command&gt; works"
else
    test_fail "&lt;command&gt; failed"
fi
```

This pattern has three critical weaknesses:

1. **No output validation**: stdout and stderr are discarded entirely. The test cannot distinguish between "command produced correct output" and "command produced garbage but exited 0".

2. **No behavioral verification**: The test does not verify that `mac system info` actually shows system information, that `mac network ip` shows an IP address, or that `mac version` shows the version string.

3. **False positive risk**: A command that silently succeeds (e.g., a no-op fallback path) would pass the test despite not performing its intended function.

For comparison, a meaningful test would look like:
```bash
output=$("$BIN_DIR/mac" version 2&gt;&amp;1)
[[ "$output" == *"SuperMac"* ]] || test_fail "Version output missing 'SuperMac'"
[[ "$output" == *"2.1.0"* ]] || test_fail "Version output missing version number"
```

#### 3.2 Test Counter Methodology

The counting system (`count_test` separate from `test_success`/`test_fail`) means the developer must remember to call `count_test` before every `test_success` or `test_fail`. If a code path has an early return or conditional that skips `count_test`, the totals will be inconsistent. There is no sanity check that `TESTS_PASSED + TESTS_FAILED == TESTS_TOTAL` at the end.

#### 5.2 `set -e` Interaction

# Distribution and Packaging Audit: SuperMac v2.1.0

## Executive Summary

SuperMac's distribution story has significant gaps that would block production adoption. The most severe issues are: 13 duplicate files tracked at the repository root (doubling repo size and creating maintenance drift), no LICENSE file despite MIT claims, no integrity verification on downloaded files, incomplete uninstall, and four library modules missing from the installer. The overall score is **38/100** -- functional for personal use but not ready for public distribution.

## 1. Install Script Quality (Score: 45/100)

### What Works

The install script (`bin/install.sh`, 357 lines) demonstrates reasonable competence in several areas. It detects the macOS platform with `sw_vers -productVersion` and warns on versions older than 12.0. It identifies the user's shell (zsh, bash, or fallback to `.profile`) and adds `~/bin` to PATH in the appropriate config file. The download function supports both curl and wget as fallbacks. A minimal fallback installation is embedded for offline scenarios.

### Critical Issues

**The download function silently swallows all failures.** `download_supermac_files()` iterates through 8 files, catches individual download failures with `print_warning`, but never tracks whether any failed. The function returns 0 (success) unconditionally. The conditional on line 276 (`if download_supermac_files; then`) always takes the success branch, making the fallback installation on line 279-281 unreachable dead code. If a user's network drops mid-download, they end up with a partial installation and a success message.

**Four modules are missing from the download list.** The installer fetches finder, display, network, system, dev, and utils -- but the project has 10 modules. Missing: `wifi.sh`, `dock.sh`, `audio.sh`, `screenshot.sh`. These are registered in `config.json` as enabled categories and exist in `lib/`, but the installer never fetches them. Users installing via `curl|bash` will have broken `mac wifi`, `mac dock`, `mac audio`, and `mac screenshot` commands.

**PATH check is brittle.** The grep on line 104 matches the exact string `export PATH="$HOME/bin:$PATH"`. It will miss alternative forms (`PATH=$HOME/bin:${PATH}`, `PATH=~/bin:$PATH`, or if the user has already configured this via `/etc/paths`). If the exact match fails, the script appends a duplicate line. It also matches commented-out lines.

**No idempotency.** Running install.sh a second time will overwrite `config.json`, destroying any user preferences (command history, favorites, last_used). It may append duplicate PATH lines. There is no version comparison to determine whether reinstalling is necessary.

## 2. curl|bash Safety (Score: 30/100)

### What Works

The curl flags (`-fsSL`) are correct: fail silently on HTTP errors, follow redirects, show errors, and use the URL from the argument. The script uses `set -e` for early termination on errors.

### Critical Issues

**No integrity verification.** The installer downloads 8 shell scripts from GitHub raw URLs and writes them directly to disk with no checksum, signature, or content validation. A man-in-the-middle attack, compromised CDN, or DNS hijack could inject arbitrary code into `~/bin/mac` which the user then executes. This is the standard criticism of the `curl|bash` pattern, and it is not mitigated here.

**Stdin conflict when piped.** The recommended install command pipes curl output into bash (`curl ... | bash`). In this mode, bash's stdin is the pipe from curl. The interactive `read -r response` on line 72 will read from the pipe (consuming subsequent lines of the install script as input) or return EOF, causing `set -e` to abort the script. On macOS versions older than 12.0, this means the installation will silently abort with no explanation.

**No audit opportunity.** The `curl|bash` pattern gives users zero chance to review the script before execution. A download-then-run pattern (`curl -fsSL URL -o /tmp/install.sh &amp;&amp; less /tmp/install.sh &amp;&amp; bash /tmp/install.sh`) would be safer.

### Recommendations

1. Generate SHA256 checksums for all distributed files during `make package`.
2. Embed checksums in the installer or fetch a checksums file alongside.
3. Verify each download with `shasum -a 256 -c` before writing.
4. Add `exec &lt; /dev/tty` before interactive prompts to reconnect stdin.
5. Offer a download-first installation option as the primary recommendation.

## 3. Packaging (Score: 35/100)

### What Works

The Makefile has well-structured `build`, `package`, and `release-check` targets. `make package` runs quality checks via the `build` dependency (which runs `check` first, running lint and syntax tests), copies relevant directories into `dist/`, and creates a tarball. `release-check` verifies that version strings match in the main script and installer.

### Issues

**Permissions may not survive packaging.** The tar command (`tar -czf ...`) does not use `--preserve-permissions`. While GNU tar preserves permissions by default for the creating user, extracting on a different system with a different umask may lose executable bits. The `cp -r` in the build step also does not preserve all metadata; `cp -a` would be more reliable.

**No MANIFEST or file list verification.** The package is created by copying directories wholesale. There is no manifest of expected files, no check that all required modules are present, and no post-build verification.

**Root duplicates contaminate the source.** While the `build` target only copies `bin/`, `lib/`, `config/`, and `docs/` (not the root duplicates), the presence of root-level copies of every file makes the repository confusing for anyone examining the source. A contributor might edit the root `finder.sh` instead of `lib/finder.sh` and not realize the changes are ignored.

**No release automation.** There is no GitHub Actions workflow, no release script, and no tag-based release process. The `release-check` target is a local manual step. Publishing a release requires manually creating a GitHub release, uploading the tarball, and updating the README.

## 4. Version Management (Score: 55/100)

### What Works

Version `2.1.0` is consistently defined in the key files. The Makefile has an `update-version` target that uses sed to update three locations. The `release-check` target verifies version consistency in `bin/mac` and `bin/install.sh`.

### Issues

**auto_update_check is a config-only fiction.** `config/config.json` declares `"auto_update_check": true`, but no code anywhere reads this setting or implements any update checking. Users who see this setting might expect the tool to notify them of updates.

**Layer 2: Version string in 9 locations**

- `lib/utils.sh:14` -- `readonly SUPERMAC_VERSION="2.1.0"`

The `install.sh` and `setup.sh` scripts also duplicate the color definitions and print helper functions from `utils.sh` (about 18 lines each). This is somewhat unavoidable since they run standalone, but extracting a minimal bootstrap file would solve it.

**Quantified impact**: Of the 7,199 total lines in the project, 5,154 (71.6%) are exact duplicates of the lib/ files. Even excluding root-level copies, the structural duplication across modules accounts for approximately 500-600 lines that could be replaced with shared utilities.

### 3. Error Handling (Score: 55/100)

# SuperMac v2.1.0 -- UX &amp; CLI Design Audit Report

## Executive Summary

SuperMac scores **72/100** for overall UX and CLI design quality. It demonstrates strong architectural consistency across its 9 modules and 73+ commands, with a clean two-tier dispatch pattern, thorough help coverage, and thoughtful error messages. The primary weaknesses are in **accessibility** (55/100) -- the tool lacks NO_COLOR support, relies on emoji for semantic meaning, and provides no way to bypass interactive prompts. **Command design** (78/100) is generally strong but undermined by inconsistent boolean parsing, hidden commands, and duplicate constants.

---

## 1. Command Design (78/100)

SuperMac uses a `mac &lt;category&gt; &lt;action&gt; [args]` pattern. This is a well-established CLI convention (used by `git`, `docker`, `npm`, `brew`) and is a good choice for a tool with 73+ commands.

**What works well:**

The category names are intuitive and map cleanly to their domains: `finder`, `wifi`, `network`, `system`, `dev`, `display`, `dock`, `audio`, `screenshot`. Action names use kebab-case consistently (`show-hidden`, `dark-mode`, `kill-port`, `disk-usage`), which is the correct convention for multi-word CLI subcommands.

The global shortcuts system is the best design decision in the entire tool. Commands like `mac ip`, `mac dark`, `mac cleanup`, `mac kp 3000`, and `mac vol 50` eliminate the two-tier nesting for the most frequent operations. These are well-chosen: they are short, memorable, and do not conflict with any standard macOS commands.

The `mac` entry point handles both `mac help &lt;category&gt;` and `mac &lt;category&gt; help` implicitly via each module's dispatcher accepting `"help"|"-h"|"--help"`. Combined with `mac -h` and `mac --help`, there are effectively four ways to reach help -- excellent for discoverability.

**Problems identified:**

Boolean argument handling is inconsistent across modules. The dock module (`dock.sh:98-100`) accepts 8 synonyms for on/off (`on/off/enable/disable/true/false/1/0`), while wifi accepts only 2 (`on/enable`, `off/disable`), and screenshot accepts 4 (`on/off/enable/disable`). This is a maintenance burden and confuses users who expect consistent behavior. A shared `parse_boolean()` function in utils.sh would solve this cleanly.

The `system` module exposes a `detailed-info` action (`system.sh:504-506`) that is not listed in the help output (`system.sh:419-449`). This is a discoverability violation. Either document it or merge it into `info` with a `--verbose` flag.

The `SUPERMAC_VERSION` constant is defined in two places (`bin/mac:43` and `lib/utils.sh:14`). This is a maintenance trap. The version should be defined once in utils.sh (which is always loaded first) and removed from bin/mac.

The `display resolution` command requires an explicit `list` sub-action but does not provide a sensible default. Running `mac display resolution` without arguments gives an error. Compare with `mac network locations` which defaults to `list` when no sub-action is provided (`network.sh:339`). The display module should follow the same pattern.

Undocumented aliases create an invisible feature layer. Commands like `mac wifi on` also accept `enable`, `mac dock position left` also accepts `l`, and `mac system memory` also accepts `mem` -- but none of these appear in help output. Users who discover one alias will expect others that may not exist.

## 2. Help System UX (82/100)

This is the strongest category. The help system is thorough, consistent, and multi-layered.

**Architecture:**

Every module implements three help-related functions: `&lt;module&gt;_help()` for detailed help, `&lt;module&gt;_search()` for keyword search, and `&lt;module&gt;_dispatch()` which handles `help/-h/--help`. The main dispatcher in `bin/mac` adds a top-level `show_main_help()`, `show_category_help()`, and `show_version()`. This is a well-structured approach.

Running `mac` with no arguments shows the main help, which is the correct behavior. The main help includes: usage syntax, category listing with emoji+descriptions, quick commands, examples, and global shortcuts. This is more comprehensive than many similar tools.

Each module's help includes: a formatted command table, usage examples, tips, and shortcuts. The `dev` module's help (`dev.sh:467-504`) is exemplary -- it includes a "Development Workflow" section that contextualizes commands for real-world use.

The search system (`bin/mac:159-192`) loads each module and calls its `_search()` function. This is a smart design that keeps search indexes close to the commands they describe.

**Weaknesses:**

The search implementation uses fragile substring matching. Each module's `_search()` function is a hand-coded if/elif chain that checks specific keywords against the search term. For example, `wifi_search()` (`wifi.sh:430-456`) checks for exact substring matches against "wifi", "wireless", "network", "connect", etc. This misses synonyms (e.g., "internet" returns nothing from wifi), fails for partial matches ("net" does not match "network" because it checks `"network" == *"net"*` which works, but "connect" won't find the wifi connect command unless you type "connect"). The box-drawing widths vary across modules (65, 70, 75, 80 characters). This is a visual inconsistency that suggests the modules were developed independently without a shared width constant.

## 10. Exit Codes (analysis)

The tool uses exit codes correctly and consistently:

- `return 0` for success (all modules).
- `return 1` for errors (all modules).
- `exit 1` only in the main dispatcher for fatal errors (missing utils.sh).
- The `die()` function (`utils.sh:448-451`) supports custom exit codes via a second argument.

The `set -euo pipefail` in `bin/mac:18` ensures that unset variables, failed commands, and pipe failures all cause the script to exit with non-zero status.

One concern: the subshell patterns used extensively (e.g., `ps aux | sort | head | while read ...`) mean that `return` inside the while loop exits the subshell, not the function. If a command inside a piped while-loop fails, the error is not propagated to the caller. This is a fundamental limitation of the Bash piped-while pattern and affects functions like `system_processes()` and `dev_list_ports()`.

# SuperMac Core Logic Audit -- Detailed Analysis

## 1. Command Routing (Score: 78/100)

The routing architecture is one of the stronger aspects of the codebase. The `main()` function in `bin/mac` implements a layered dispatch system:

**Layer 1 -- Special commands** (lines 301-336): `help`, `version`, `search`, `debug` are handled directly via a `case` statement before any module loading occurs. This is efficient because it avoids loading modules for commands that do not need them.

**Layer 2 -- Global shortcuts** (lines 224-243): The `route_command()` function checks `GLOBAL_SHORTCUTS` (an associative array) before validating the category. Shortcut values use a `category:action` format parsed via `IFS=':' read`. This is clean and extensible.

**Layer 3 -- Module dispatch** (lines 247-277): Category is validated against `CATEGORIES`, the module is loaded via `load_module()`, and then `{module}_dispatch` is called with the action and remaining arguments.

The main routing weakness is the `debug` command handler (lines 323-335), which recursively calls `main()`. While this works, it re-runs `validate_environment()` and could cause subtle issues if combined with other flags. A cleaner approach would be to strip the `--debug` flag during initial argument parsing.

Another minor issue: the `search_commands()` function (lines 159-192) loads every single module during a search operation. Since `load_module()` has no re-sourcing guard, this re-defines all functions in every module for each search. For 9 modules this is acceptable performance-wise, but it is architecturally sloppy.

The routing correctly handles:
- Unknown categories with helpful error messages
- Missing actions with category-specific help suggestions
- Unknown actions within modules with available-action listings
- Special flags (`-h`, `--help`, `-v`, `--version`)

## 2. Module Loading (Score: 58/100)

The module loading system has two significant problems.

**Problem 1: No re-sourcing guard.** The `load_module()` function in `lib/utils.sh` (lines 510-521) unconditionally sources the target module file. Every call to `load_module("system")` re-executes the entire `lib/system.sh` script, redefining all functions and executing the trailing `print_debug` statement. During search operations, all 9 modules are sourced. This works because Bash function definitions are idempotent (re-defining a function simply overwrites it), but it is wasteful and could cause issues if modules ever have side effects at source time beyond function definitions.

A simple fix would be:
```bash
declare -gA _LOADED_MODULES 2&gt;/dev/null || true
load_module() {
    local module_name="$1"
    [[ -v _LOADED_MODULES[$module_name] ]] &amp;&amp; return 0
    _LOADED_MODULES[$module_name]=1
    source "$(get_supermac_dir)/lib/${module_name}.sh"
}
```

**Problem 2: Dual path resolution.** `bin/mac` computes `LIB_DIR` from `SCRIPT_DIR` (line 27), but `load_module()` in `utils.sh` independently computes the module path using `get_supermac_dir()` (which resolves from `BASH_SOURCE[0]`). These two path computations start from different anchors and could diverge. During development from the repo, both happen to resolve to the same directory. But after installation via `install.sh`, `bin/mac` lives at `~/bin/mac` while `utils.sh` lives at `~/.supermac/lib/utils.sh`. The `bin/mac` hardcodes `LIB_DIR` relative to its own location, but then delegates to `load_module()` which resolves relative to `utils.sh`. These should agree, but the dual resolution is a maintenance risk.

**Problem 3: `readlink -f` is not portable to macOS.** The `get_supermac_dir()` function uses `readlink -f`, which is a GNU coreutils extension not available on stock macOS. On macOS, `readlink` does not support `-f`. The `2&gt;/dev/null` suppression hides the error, and the fallback to `${BASH_SOURCE[0]}` works for non-symlinked installations. But if SuperMac were ever installed via symlink, the path resolution would break silently.

**The vm_stat page size bug** deserves attention. In `lib/system.sh` lines 266-281:
```bash
local page_size=4096
pages_free=$(echo "$memory_info" | grep "Pages free" | awk '{print $3}' | sed 's/\.//')
free_mb=$((pages_free * page_size / 1024 / 1024))
```
On Apple Silicon (M1/M2/M3/M4), the kernel page size is 16384 bytes, not 4096. The correct value can be obtained from `sysctl -n hw.pagesize`. This means all memory statistics displayed by `mac system memory` are understated by a factor of 4 on Apple Silicon machines -- the most common modern Macs.

**The airport path issue** means that `mac wifi scan`, `mac wifi info`, and `mac wifi status` never display signal strength, SSID details, channel information, or BSSID data. The `command_exists airport` check always fails because the airport binary is not on the default PATH. Users see the fallback behavior (listing preferred networks or basic connection info) without knowing they are missing features.

## 4. Configuration System (Score: 20/100)

This is the weakest area of the codebase. The configuration system is completely non-functional at runtime:

- `config/config.json` defines `default_settings` for `screenshot_location`, `screenshot_format`, `volume_step`, `brightness_step`, `color_output`, `debug_mode`, and `auto_update_check`.
- `lib/utils.sh` provides `get_config()` to read these values via `jq`.
- **No module ever calls `get_config()`.** Not a single invocation exists in the entire codebase.

Every module hardcodes its own defaults:
- `screenshot_get_location()` defaults to `$HOME/Desktop` (ignoring config's `screenshot_location: "Desktop"`)
- `audio_volume_up()` defaults step to `10` (ignoring config's `volume_step: 10`)
- `display_set_brightness()` has no step concept (ignoring config's `brightness_step: 10`)

The `aliases` section of config.json duplicates `GLOBAL_SHORTCUTS` in `bin/mac` but is never read. The `user_preferences` section with `favorite_commands`, `command_history`, and `last_used` is completely unused. The `categories` section with `enabled` flags is also unused -- all categories are always available regardless of the config.

The config system appears to have been designed as a future feature that was never integrated. As it stands, it is misleading: users who edit config.json to change settings will find their changes have no effect.

Additionally, `get_config()` requires `jq` to be installed, but this dependency is not documented or checked at startup. If `jq` is missing, `get_config()` silently returns the default value, which is reasonable but undocumented.

## 5. Data Flow Integrity (Score: 60/100)

Tracing a complete request reveals several concerns:

**Example trace: `mac system info`**

1. `bin/mac:347` -- `main()` is called with `["system", "info"]`
2. `bin/mac:286` -- `validate_environment()` checks macOS and lib directory existence
3. `bin/mac:339` -- `route_command("system", "info")` is called
4. `bin/mac:256` -- `load_module("system")` sources `lib/system.sh`
5. `bin/mac:270` -- `system_dispatch("info")` is called
6. `lib/system.sh:27-78` -- `system_info()` executes

Within `system_info()`, there are 4 calls to `system_profiler SPHardwareDataType` (lines 42-44), each spawning a new process and taking 200-500ms. The total latency for this single command can exceed 2 seconds. These should be consolidated into a single call with cached output:

```bash
local hw_data
hw_data=$(system_profiler SPHardwareDataType 2&gt;/dev/null)
model_name=$(echo "$hw_data" | grep 'Model Name' | ...)
chip_info=$(echo "$hw_data" | grep 'Chip' | ...)
memory_info=$(echo "$hw_data" | grep 'Memory' | ...)
```

**The `set -euo pipefail` concern**: Multiple modules use pipe patterns like `ps aux | sort -nr -k 3 | head -10 | while read ...`. Under `pipefail`, if `head` closes the pipe early (which it does after reading 10 lines), the preceding `sort` gets SIGPIPE and exits non-zero. Under `set -e`, this should abort the script. The reason it currently works is that these pipelines run inside the `while read` loop's implicit subshell, and the exit code of the pipeline is not checked by the enclosing function. However, this is fragile and any change to the error handling strategy could cause breakage.

**The screenshot_take() data flow bug**: `screenshot_take()` at `lib/screenshot.sh:400-433` saves screenshots to `/tmp/screenshot-$(date +%s).png` regardless of the configured screenshot location. After taking the screenshot, it prints "Screenshot saved to: $location" where `$location` is the configured location, not `/tmp`. The file is actually in `/tmp` but the user is told it is in the configured location. This is a clear data flow disconnect.

## 6. Edge Case Handling (Score: 50/100)

**Dangerous `/tmp/*` deletion**: `system_cleanup()` at `lib/system.sh:173` runs `rm -rf /tmp/*`. This is the most dangerous operation in the codebase. `/tmp` contains Unix sockets for running applications, PID files, temporary databases (SQLite WAL files for browsers), and named pipes. Deleting all of these can crash running applications or corrupt data. The `|| true` suppression hides any errors, giving the user false confidence. The fix is straightforward: use `find /tmp -type f -atime +7 -user $(whoami) -delete` to only clean old files owned by the current user.

**Floating-point comparison inconsistency**: `dev_cpu_hogs()` uses string comparison for numeric thresholds (`[[ "$cpu" &gt; 5.0 ]]`), while `dev_memory_hogs()` correctly uses `bc -l` for the same purpose. The string comparison fails for values &gt;= 10 because lexicographic ordering puts "10.0" before "5.0" (since "1" &lt; "5").

**No IPv6 support**: `network_get_local_ip()` only returns IPv4 addresses via `ipconfig getifaddr`. On networks with IPv6-only connectivity, this returns nothing and the user is told "No active network connection found" despite having working IPv6 connectivity.

**Battery parsing on desktop Macs**: `system_battery()` checks for SPPowerDataType availability but the check (`system_profiler SPPowerDataType &gt;/dev/null 2&gt;&amp;1`) succeeds even on desktop Macs, returning data about AC power. The subsequent parsing of battery-specific fields (percentage, cycle count) may return empty strings, leading to incomplete output rather than a clear "no battery" message.

**sudo commands without pre-check**: `network_flush_dns()` at `lib/network.sh:199-216` runs `sudo dscacheutil -flushcache` without checking if the user has sudo access. If the user does not have admin privileges, the command will prompt for a password in the middle of the operation, which is a poor user experience for a command that should check preconditions upfront.

**The `find` command in `get_available_modules()`** at `lib/utils.sh:524-529` uses `find` to list modules. If the `lib/` directory ever contains non-module files (like `.DS_Store`, `__pycache__`, etc.), they would be listed as available modules. The `-name "*.sh" -not -name "utils.sh"` filter is good but could be stricter.

**Memory page size hardcoding** as mentioned in the macOS API section causes all memory statistics to be wrong on Apple Silicon. This is the most impactful edge case since M-series Macs are now the majority of the installed base.

## Summary

SuperMac v2.1.0 demonstrates solid architectural design in its modular dispatcher pattern and consistent module interface. The routing system is well-structured with good error messages and global shortcuts. However, the implementation has several significant gaps:

The codebase is well-intentioned and mostly functional for common use cases. The user-facing experience (color output, helpful tips, confirmation prompts) is polished. # Bug - Local Bug Tracker

Record bugs without leaving the terminal. Uses `ccs issues create` for deterministic IDs.

## Usage

```bash
/bug "title"              # Create new bug
```

For listing bugs, use `/issues`.

## Instructions

### Step 1: Get Bug Details

Ask user for required and optional fields:

| Field | Required | Options |
|-------|----------|---------|
| Description | **Yes** | What's the problem? Expected vs actual? |
| Severity | **Yes** | critical \| high \| medium \| low |
| Fix Direction | **Yes** | How to fix it (even if preliminary) |
| Affected Files | **Yes** | Comma-separated file paths |
| Component | No | Which part of the system? |
| Area | No | security \| performance \| ux \| reliability \| infra |

### Step 2: Create Bug

**All 4 required flags must be provided** or CCS will reject with `<FILL>` placeholders:

```bash
ccs issues create bug "title" \
  --description "What's broken, expected vs actual behavior" \
  --severity {critical|high|medium|low} \
  --fix-direction "How to fix: approach, steps, or direction" \
  --affected-files "path/to/file1.go,path/to/file2.go" \
  [--component ...] \
  [--area ...]
```

Use `--bare` to skip validation for quick one-liners.

### Step 3: Enrich the Issue (mandatory)

After `ccs issues create` outputs the bug ID and file path, **always** enrich the issue by editing the YAML file.

Read the created file and add context from the current conversation:

```yaml
# Always add if known:
affected_files:
  - path/to/file.go
fix_direction: |
  What approach should be taken (even if preliminary)
session_ref: "Session context where this was discovered"

# Add if reproduction steps are known:
reproduction_steps: |
  1. Exact steps to reproduce
actual_behavior: "What happens"
expected_behavior: "What should happen"

# Add for complex bugs:
root_cause: "If you've identified it"
scope_boundaries: |
  In scope: X
  Out of scope: Y
```

**Never skip this step.** Even minimal context helps future implementers.

### Step 4: Output

Report the created bug ID to the user.
